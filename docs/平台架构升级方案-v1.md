# 喜事达工具平台 · 架构升级方案 v1

> 目标：把当前"少量工具、手工部署"的站点，升级为可长期承载**大量工具、多人贡献、内外双用途**的工具平台。
> 本文是设计蓝图，经确认后分三阶段实施。

---

## 1. 背景与现状

bot.regs.com 目前已上线门户首页 + 4 个工具（AEO 合规平台、装箱软件、危化判定、出口研判），并有"上传/更新工具"提交入口（进待审核暂存区）。

现状的几个不利于扩展的点：

- 工具以 HTML 文件**平铺在站点根目录**，靠维护者 SSH 手工部署；工具一多就混乱。
- 首页工具清单是**手工编辑**的 `TOOLS` 数组，每加一个工具都要改代码。
- **内外不分**：所有工具对公网可见，没有"仅内部"的概念。
- 审核靠**命令行**，没有网页后台；无版本管理、无回滚。

随着上传的同事越来越多，这些都会成为瓶颈。

## 2. 设计目标与原则

1. **数据驱动**：工具信息进数据库，首页与各视图自动渲染，新增/改版不改代码。
2. **内外分层**：每个工具可标记面向"内部 / 外部 / 通用"，并在服务端强制访问控制（不仅是前端隐藏）。
3. **自助提交、受控发布**：同事自助上传，维护者在网页审核台一键批准即自动发布，全程留痕、可回滚。
4. **隔离与安全**：工具各自独立目录；内部工具与合规数据严格隔离；上传严格加固。
5. **平滑演进**：分阶段落地，每阶段独立可用，不影响现有线上工具。
6. **统一体验**：所有工具沿用《前端统一标准》（cstar-ui.css + 功能基线）。

## 3. 总体架构

```
                      bot.regs.com (nginx)
   ┌─────────────────────────┬───────────────────────────────┐
   │  /            门户首页    │  从 /api/catalog 拉取工具清单     │
   │  /cstar-ui.css 共享样式   │  （按访问者身份过滤内部/外部）      │
   │  /tools/<slug>/  各工具   │  ← 审核通过后自动发布到此目录       │
   │  /api/*       后端接口    │  catalog / submit / admin        │
   │  /aeo/        合规平台    │  内部应用（已隔离）                │
   │  /admin/      管理后台    │  审核台 + 工具管理（仅维护者）      │
   └─────────────────────────┴───────────────────────────────┘
                              │
                    FastAPI + SQLite + 数据卷
        Tool / ToolVersion / Submission / User&Role / AuditLog
        工具文件： /data/tools/<slug>/<version>/  (版本化留存)
```

核心思想：**注册表（数据库）是唯一事实来源**。首页、客户视图、访问控制、发布、回滚，全部围绕注册表运转。

## 4. 核心数据模型

在现有 FastAPI + SQLite 上扩展。

### Tool（工具）
| 字段 | 说明 |
|------|------|
| id | 主键 |
| slug | 唯一英文标识，决定 URL `/tools/<slug>/` |
| name | 显示名称 |
| category | 分类（报关/物流/合规/财务/通用…） |
| summary | 一句话说明 |
| owner_dept | 负责人/部门 |
| icon / color | 卡片图标字/主题色（取自统一调色板） |
| **visibility** | `external` 对外 / `internal` 仅内部 / `both` 通用 |
| status | `online` 上线 / `hidden` 隐藏 / `deprecated` 已退役 |
| entry_kind | `static`（/tools/<slug>/）/ `proxy`（如 AEO 指向 /aeo/） |
| current_version_id | 指向当前生效版本 |
| created_at / updated_at | |

### ToolVersion（工具版本）
| 字段 | 说明 |
|------|------|
| id | 主键 |
| tool_id | 所属工具 |
| version | 版本号，如 v1.2.0 |
| path | 该版本文件目录 `/data/tools/<slug>/<version>/` |
| changelog | 更新说明 |
| submitted_by / approved_by | 提交人 / 批准人 |
| approved_at | 批准时间 |
| state | `pending` 待审 / `live` 生效 / `archived` 历史 |

### Submission（提交，已存在，略作扩展）
保留现有字段（token/kind/target/name/version/developer/desc/selfcheck/size/status…）。审核**批准**时，由它生成/更新对应的 Tool 与 ToolVersion，并把暂存 zip 解压发布。

### 角色（基于现有 RBAC 扩展）
| 角色 | 能力 |
|------|------|
| 访客（匿名） | 仅浏览/使用**外部**工具 |
| 外部客户 customer | 登录后使用外部工具（合规数据已隔离，不可见） |
| 内部员工 staff（各部门角色） | 浏览/使用**全部**工具 |
| 维护者 maintainer | 在管理后台审核、发布、回滚、改工具信息 |
| 管理员 admin | 全部 + 用户与角色管理 |

## 5. 可见性与访问控制（关键）

可见性必须在**服务端强制**，不能只靠前端不显示卡片——否则知道直链就能访问内部工具。

- **目录接口过滤**：`GET /api/catalog` 根据请求者身份返回可见工具（匿名/客户只得 external；员工得全部）。首页据此渲染。
- **内部工具文件门禁**：`/tools/<slug>/` 中标为 internal 的工具，其静态文件请求要经过鉴权。两种实现（实施时择一）：
  - 方案 A（推荐）：内部工具改由后端带鉴权地分发（`/api/tool-file/<slug>/...`，校验 staff 后返回文件）。
  - 方案 B：nginx `auth_request` 调后端校验子请求，通过才放行 `/tools/<slug>/`。
- 外部工具（external/both）维持 nginx 直接静态服务，性能好。

## 6. 提交 → 审核 → 发布 流水线

状态机：

```
同事上传 zip ──► [待审核] ──审核台批准──► 解压到 /data/tools/<slug>/<新版本>/
                    │                       更新 Tool.current_version_id
                    │                       旧版本转 archived（保留，可回滚）
                    └──驳回(附原因)──► [已驳回]（通知提交人）
```

- **新工具**：批准时创建 Tool（设 slug/visibility/category 等）+ 首个 ToolVersion，并写入注册表、上线。
- **更新现有工具**：批准时新增 ToolVersion、切换 current 指针；旧版保留。
- **回滚**：把 current 指回任一历史版本即可，一键完成。
- 全程写 AuditLog（已具备）。

## 7. 托管与目录结构

- 工具文件：`/data/tools/<slug>/<version>/`（版本化留存于数据卷）。
- 对外路径：`/tools/<slug>/` → 指向该工具 current 版本（用符号链接或后端按注册表解析）。
- 首页弹窗仍用 `?embed=1` 内嵌；工具继续遵循《前端统一标准》。
- `cstar-ui.css`、`tool-template.html` 继续放根目录共享。
- AEO 等"应用型"工具用 `entry_kind=proxy` 指向其现有路径（如 `/aeo/`），不强行塞进 `/tools/`。

## 8. 接口设计（概要）

| 方法/路径 | 鉴权 | 说明 |
|-----------|------|------|
| `GET /api/catalog` | 公开（按身份过滤） | 返回当前可见工具列表，首页渲染用 |
| `POST /api/tools/submit` | 公开（已加固） | 上传新工具/更新版本到暂存区（已实现） |
| `GET /api/admin/submissions` | 维护者 | 待审列表 |
| `POST /api/admin/submissions/{id}/approve` | 维护者 | 批准并发布（建/更新 Tool+Version、解压、切 current） |
| `POST /api/admin/submissions/{id}/reject` | 维护者 | 驳回+原因 |
| `GET/PUT /api/admin/tools[/{id}]` | 维护者 | 工具信息维护（名称/分类/可见性/状态/退役） |
| `POST /api/admin/tools/{id}/rollback` | 维护者 | 回滚到指定历史版本 |

## 9. 管理后台（/admin/，仅维护者）

- **审核台**：待审提交列表（类型/目标工具/版本/开发者/自检状态/大小），可下载预览 zip、批准（填 slug/分类/可见性）、驳回（填原因）。
- **工具管理**：所有工具一览，编辑信息、切换可见性（内部/外部/通用）、隐藏/退役、查看版本历史与一键回滚。
- 退役命令行操作，维护者全程在网页完成。

## 10. 现有工具迁移

| 工具 | slug | entry_kind | 建议可见性 |
|------|------|-----------|-----------|
| AEO 认证管理平台 | aeo | proxy → /aeo/ | internal（内部+授权客户演示） |
| Cstar 装箱软件 | packer | static/proxy → /packer/ | both 通用 |
| 危化判定（MSDS/UN） | msds-un | static | both 通用 |
| 出口海关研判助手 | export-customs | static | both 通用 |

迁移只是把它们登记进注册表并设可见性，**现有 URL 保持可用**，平滑过渡。

## 11. 分阶段路线图

### 阶段一：注册表 + 内外可见性 + 首页自动渲染（基础）
- 建 Tool/ToolVersion 模型与迁移；`GET /api/catalog`（按身份过滤）。
- 首页改为从 catalog 接口渲染（替换手工 TOOLS 数组）。
- 内部工具加服务端门禁（方案 A 或 B）。
- 迁移现有 4 个工具进注册表。
- **交付**：首页由数据库驱动；内部/外部分层生效；新增工具不再改首页代码。

### 阶段二：网页审核台 + 自动发布 + 版本化回滚（流水线）
- `/admin/` 审核台与工具管理页；approve/reject/rollback 接口。
- 批准即解压发布到 `/tools/<slug>/<version>/`、切 current、留历史。
- **交付**：维护者全程网页操作，退役命令行；可回滚。

### 阶段三：完善（治理与体验）
- 外部客户账号对 external 工具的访问控制（按需）。
- 首页分类/搜索；使用量统计；工具退役与下线流程。
- **交付**：面向规模化的治理能力与更好的浏览体验。

每阶段独立上线、互不阻塞，且不影响现有线上工具。

## 12. 安全与治理

- 上传：限大小/仅 zip/校验文件头/随机存储/不自动解压（已实现）；发布时再做 zip 安全解压（防路径穿越/zip 炸弹）。
- 可见性服务端强制；内部工具直链也鉴权。
- 合规数据（AEO）与工具平台严格隔离，外部角色不可见。
- 全程审计日志：谁提交、谁批准/驳回/回滚、何时。
- 版本留存便于追溯与回滚。

## 13. 与《前端统一标准》的衔接

注册表不替代前端标准，二者配合：标准保证**单个工具**长得统一、功能达标；平台保证**多个工具**的目录、内外分层、审核发布、版本治理。审核台批准前，维护者仍按《前端统一标准》部署前清单核对。
